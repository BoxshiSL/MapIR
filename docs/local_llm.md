# Local LLM Drafting

MapIR v0.4 ships a small **local** LLM drafting layer. You describe what you
want (a city, a port, an alley, a warehouse interior); a local model produces
a high-level **Plan**; MapIR's deterministic code converts the plan into
validator-friendly **WorldIR / SceneIR**; and your existing validators decide
whether the draft is ever shown as `OK`.

The LLM is an **assistant**, not a source of truth. The source of truth stays:

- MapIR schemas (`mapir/core/models.py`)
- Pydantic structural validation
- Semantic validators (`mapir/core/validation.py`)
- Deterministic Plan → IR conversion (`mapir/llm/plan_to_ir.py`)
- Your review

Invalid drafts are never silently accepted.

## What it does

- Drafts a **WorldPlan** from a natural-language brief, converts to **WorldIR**.
- Drafts a **ScenePlan** from a brief, converts to **SceneIR**.
- Drafts a **DistrictProfile** for an existing world.
- Repairs an invalid document by feeding the validation errors back to the model.
- Runs validation after every step and reports problems to the UI / CLI.

## What it does NOT do

- It does **not** download models. You install models yourself via Ollama.
- It does **not** call cloud APIs. There is no OpenAI / Claude / Gemini path.
- It does **not** send your files outside your machine — only local Ollama is
  used by default.
- It does **not** ask the model to invent precise coordinates. Geometry is
  generated by deterministic helpers from the plan.
- It does **not** auto-overwrite your current document. You explicitly click
  *Load Draft as Current Document* (or save the JSON manually).

## Ollama setup

1. Install Ollama: <https://ollama.com>.
2. Start the Ollama service (the installer typically does this for you).
3. Pull a model — see "Recommended models" below.
4. In MapIR's *LLM Draft* page, set the model name and click *Check Provider*.

The default base URL is `http://127.0.0.1:11434`. Override it under
*Provider → Base URL* if your daemon is bound elsewhere.

## Recommended models

Local model choice is **catalog-dependent** — model availability and exact
tags vary across Ollama versions. The names below are common defaults; check
`ollama list` and the Ollama library before pulling.

Good starting points:

- **Qwen** family — strong at structured JSON drafting:
  - `ollama pull qwen3:8b`
  - `ollama pull qwen3:14b`
  - `ollama pull qwen2.5:7b-instruct`
  - `ollama pull qwen2.5:14b-instruct`
- **DeepSeek-R1-Distill** — reasoning-heavy drafts:
  - `ollama pull deepseek-r1:7b`
  - `ollama pull deepseek-r1:14b`

The default in `settings/mapir_settings.json` is `qwen3:8b` with
`deepseek-r1:7b` as a fallback hint. Adjust to whatever you have locally.

## Structured JSON output

When *Structured output* is enabled (default), MapIR sends a JSON schema to
Ollama via the `format` field. Models that support it (recent Qwen, Llama,
DeepSeek builds) return strict JSON directly. Models that ignore `format`
still get the JSON-only instruction in the system prompt — and MapIR strips
any stray Markdown fences before parsing.

If you see frequent JSON-parse errors, try:

1. Lower temperature (`0.0–0.2`).
2. A larger / more recent model.
3. Pull the model again — Ollama sometimes updates tag definitions.

## Repair loop

When the converted IR fails validation, MapIR can ask the model to repair the
JSON given the list of validator errors. This is bounded by
`max_repair_attempts` (default `1`). Repaired output is **re-parsed and
re-validated** before it is shown as `OK`. If the repaired output still fails,
the panel shows the latest errors so you can adjust the brief or try again.

## Troubleshooting

### Ollama not running

> `UNAVAILABLE provider=ollama` / `Cannot reach Ollama at http://127.0.0.1:11434`

Start the Ollama desktop app (or service). Then click *Check Provider* again.

### Model not found

> `Model 'qwen3:8b' not found. Run: ollama pull qwen3:8b`

Pull the model with that exact tag (case-sensitive). Use `ollama list` to see
what you actually have installed.

### Model returns invalid JSON

> `Model did not return valid JSON: ...`

- Lower temperature.
- Try a different model — some smaller distilled variants struggle with
  schema adherence.
- The repair loop will often recover from a single bad response; if it
  doesn't, simplify the brief.

### Generation too slow

CPU-only inference of a 7-14B model is genuinely slow (minutes per draft on
a laptop CPU). Options:

- Use a smaller model (`qwen3:4b`, `qwen3:8b` instead of 14B).
- Enable GPU offload in Ollama (`OLLAMA_NUM_GPU=...`).
- Increase `Timeout (s)` in the panel; default is 120s.

### Low RAM / VRAM

If Ollama crashes mid-generation, the model is too large for your machine.
Pick a smaller tag or a Q4-quantised build.

### Timeout

> `Ollama timed out after 120s`

Increase the timeout in the panel or in `settings/mapir_settings.json`
(`timeout_seconds`). Default 120s is enough for `qwen3:8b` with a warm cache
and a GPU, but not for cold-start CPU runs.
